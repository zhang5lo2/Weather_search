# AI 训练模型浓缩指导大纲

本文根据《机器学习训练秘籍（Machine Learning Yearning）》整理，目标是给 AI/机器学习模型训练提供一份简洁的实战决策大纲。

## 1. 总原则

- 不要一开始就盲目调模型、堆参数、换架构。
- 先明确评估目标，再训练模型。
- 先做可用基线，再通过误差分析决定下一步。
- 每次只改变一个主要因素，保留实验记录。
- 训练 AI 模型的关键不是“试更多方法”，而是“知道该优先试什么”。

## 2. 建立正确的数据划分

### 训练集

用于模型学习参数，可以尽量大。

### 开发集

也叫验证集，用于调模型、选超参数、比较实验结果。

### 测试集

只用于最终评估，不应频繁查看或参与调参。

### 核心要求

- 开发集和测试集应来自相同分布。
- 开发集和测试集应代表真实应用场景。
- 训练集可以和开发/测试集不完全同分布，但必须知道这种差异。
- 大数据项目不必机械使用 `70% / 30%` 划分，应按评估需要设置开发集和测试集规模。

## 3. 先定义评估指标

训练前要先确定一个主要指标，否则模型迭代会失去方向。

常见主指标：

- 分类任务：accuracy、macro F1、balanced accuracy。
- 检测任务：mAP、recall、precision。
- 回归任务：MAE、RMSE、R2。
- 生成任务：人工评分、任务成功率、偏好胜率。

如果存在多个目标，应区分：

- 优化指标：主要追求提升的指标。
- 满意度指标：必须达到最低要求的约束。

示例：

```text
优化指标：准确率最高
满意度指标：推理时间 < 100ms，模型大小 < 50MB
```

## 4. 快速建立基线模型

第一版模型不需要完美，目标是尽快跑通完整流程。

基线模型应包含：

- 固定数据划分。
- 固定评估指标。
- 可复现的训练脚本。
- 训练日志。
- 验证集结果。
- 错误样本导出。

基线模型的价值是提供比较对象。后续所有优化都应回答：是否比基线更好，为什么更好。

## 5. 做误差分析

不要只看总准确率。应抽取模型在开发集上的错误样本，人工查看错误原因。

推荐流程：

1. 收集约 100 个错误样本。
2. 标记每个错误属于哪种类型。
3. 统计各类错误占比。
4. 优先解决占比最高、最容易改进的错误。

常见错误类型：

- 标签错误。
- 样本质量差。
- 类别边界模糊。
- 训练数据不足。
- 真实场景与训练数据不匹配。
- 模型能力不足。
- 评估指标不符合实际目标。

误差分析的目的不是解释结果，而是决定下一步行动。

## 6. 处理标签错误

标签错误不一定总是最优先处理，取决于它是否严重影响评估。

判断方法：

```text
标签错误占错误样本比例 = 标签错误数量 / 总错误数量
```

如果标签错误占比很小，可以先记录，暂不处理。

如果标签错误占比较大，尤其影响模型比较结果，应优先清洗开发集和测试集。

注意：

- 修正开发集标签时，也要用同样标准修正测试集。
- 不要只清洗模型预测错的样本，否则可能引入偏差。
- 训练集存在少量标签噪声通常可以容忍，但开发/测试集标签质量更关键。

## 7. 判断偏差和方差

训练模型时，先判断问题来自哪里。

| 现象 | 问题 | 优先措施 |
|---|---|---|
| 训练集差，开发集也差 | 高偏差 | 更大模型、训练更久、改架构、减少正则化 |
| 训练集好，开发集差 | 高方差 | 更多数据、数据增强、正则化、早停 |
| 训练集和同分布验证集好，真实开发集差 | 数据不匹配 | 收集更贴近真实场景的数据 |
| 人也难判断 | 任务/标签定义问题 | 明确类别标准或调整任务定义 |

简单判断公式：

```text
可避免偏差 ≈ 训练误差 - 人类水平或理论最优误差
方差 ≈ 开发误差 - 训练误差
数据不匹配 ≈ 目标开发集误差 - 训练同分布验证误差
```

## 8. 降低高偏差

当模型在训练集上都表现不好时，优先降低偏差。

可尝试：

- 使用更大模型。
- 训练更久。
- 改进模型架构。
- 使用预训练模型。
- 提高输入分辨率或特征质量。
- 减少过强正则化。
- 根据训练集错误样本补充特征或数据。

不优先做：

- 盲目增加大量同类数据。
- 过早加强正则化。
- 只调小学习率但不分析模型是否有能力拟合。

## 9. 降低高方差

当训练集表现好、开发集表现差时，优先降低方差。

可尝试：

- 增加训练数据。
- 数据增强。
- L2 正则化。
- Dropout。
- Early stopping。
- 减小模型复杂度。
- 清洗异常训练样本。
- 使用交叉验证或更稳定的数据划分。

通常最可靠的方法是增加与目标场景一致的数据。

## 10. 处理数据不匹配

如果训练数据和真实应用场景不同，模型可能在训练集表现很好，但上线效果差。

建议额外建立：

```text
train-dev 集
```

它来自训练集同分布，但不参与训练。

对比方式：

| 结果 | 解释 |
|---|---|
| train 差 | 偏差问题 |
| train 好，train-dev 差 | 方差问题 |
| train-dev 好，dev 差 | 数据不匹配问题 |
| dev 好，test 差 | 过拟合开发集或测试集分布不同 |

解决数据不匹配：

- 收集更接近真实场景的数据。
- 将真实场景数据加入开发集和测试集。
- 对训练数据做针对性增强。
- 使用迁移学习或领域适应方法。
- 不要只在旧分布数据上追求高分。

## 11. 使用学习曲线

学习曲线用于判断加数据、换模型、调正则化是否值得。

做法：

```text
用 25%、50%、75%、100% 训练集分别训练模型
记录训练误差和开发误差
```

解读：

- 训练误差高，开发误差也高：偏差高。
- 训练误差低，开发误差高：方差高。
- 增加数据后开发误差持续下降：继续加数据有价值。
- 增加数据后提升很小：可能需要改模型、改标签或改任务。

## 12. 使用人工合成数据

人工合成数据有用，但必须模拟真实错误分布。

适合使用的情况：

- 真实场景中某类样本缺少。
- 错误集中在可模拟的条件上。
- 合成方式不会破坏标签含义。

风险：

- 合成数据看似真实，但模型学到伪特征。
- 合成分布过窄，导致模型只适应某种固定模式。
- 增强过强，反而让任务语义变混乱。

原则：

```text
先通过误差分析发现缺什么，再合成什么。
```

## 13. 端到端学习与流水线

端到端学习适合：

- 输入和输出之间有大量标注数据。
- 任务边界清晰。
- 模型有足够数据学习中间规律。

流水线适合：

- 数据量较小。
- 中间步骤容易人工定义。
- 错误需要拆解定位。
- 某些组件可以单独优化。

选择原则：

- 数据足够大时，端到端通常更有潜力。
- 数据不足时，合理的流水线和人工知识更可靠。
- 如果系统复杂，应按组件做误差分析，找出最值得优化的部分。

## 14. 实验迭代流程

推荐每轮训练按以下顺序执行：

1. 明确本轮要验证的问题。
2. 固定数据划分和主指标。
3. 只改变一个主要变量。
4. 训练模型。
5. 在开发集评估。
6. 导出错例。
7. 做误差分析。
8. 记录实验结果。
9. 决定下一轮方向。
10. 最后才在测试集评估。

## 15. 实验记录模板

| 实验名 | 改动 | 主指标 | 训练集表现 | 开发集表现 | 主要错误 | 结论 |
|---|---|---:|---:|---:|---|---|
| baseline | 初始模型 | | | | | |
| exp01 | 更强增强 | | | | | |
| exp02 | 更大模型 | | | | | |

每个实验必须能回答：

- 改了什么？
- 指标是否提升？
- 错误类型是否改善？
- 是否值得保留？

## 16. 调优优先级清单

训练 AI 模型时，建议按以下优先级排查：

1. 数据划分是否正确。
2. 指标是否符合真实目标。
3. 标签是否可靠。
4. 错误主要集中在哪里。
5. 是偏差问题还是方差问题。
6. 是否存在数据不匹配。
7. 是否需要更多数据。
8. 是否需要更强模型。
9. 是否需要正则化或增强。
10. 是否需要重新定义任务。

## 17. 最简行动准则

- 没有开发集，不开始调参。
- 没有单一主指标，不比较模型。
- 没有基线，不谈优化。
- 没有错例分析，不决定下一步。
- 训练集差，先解决偏差。
- 训练集好但开发集差，先解决方差。
- 真实场景差，先解决数据不匹配。
- 标签混乱，先修任务定义和标注标准。
- 小幅指标提升要谨慎，尤其在小开发集上。
- 最终测试集只用于最终确认，不用于反复选择模型。

## 18. 一句话总结

训练 AI 模型的有效路线是：先用开发集和指标确定方向，再用基线模型暴露问题，通过误差分析判断偏差、方差、数据不匹配或标签问题，最后有针对性地改数据、改模型、改增强和改评估。
